# Yes-And: ftux-goal-arc-onboarding

Original idea: first-run onboarding should use the concrete-to-identity flow to create both a Goal and an Arc, then land the user on the Arc.

**Yes, and what if it could...** teach the object model by creating it, not explaining it?

- Serves: `jtbd-trust-this-app-with-my-life`
- Job elevation: The user sees Goal inside Arc as product behavior, not terminology.
- New value: FTUX can avoid an explainer-heavy "what is an Arc?" moment.
- Cost delta vs. original: low
- Anti-pattern check: pass; task-first orientation before noun-first terminology.

**Yes, and what if it could...** make the first Goal the first evidence promise of the Arc?

- Serves: `jtbd-see-who-im-becoming`
- Job elevation: The Arc is immediately actionable because it has a linked Goal.
- New value: The app does not start with an abstract identity card that has no traction.
- Cost delta vs. original: medium
- Anti-pattern check: pass if the Goal remains concrete and the Arc remains identity-shaped.

**Yes, and what if it could...** land on the Arc with the Goal visible as the first expression?

- Serves: `jtbd-name-my-arcs`
- Job elevation: The user receives the identity payoff and sees the concrete next move in context.
- New value: Landing reinforces the hierarchy: Arc gives meaning, Goal gives traction.
- Cost delta vs. original: low to medium
- Anti-pattern check: pass; avoid dashboard or checklist-first landing.

**Yes, and what if it could...** keep direct Arc creation as an expert path outside FTUX?

- Serves: `jtbd-name-my-arcs`
- Job elevation: Users who already know their identity direction are not forced through concrete-first input later.
- New value: FTUX gets simpler without removing advanced creation capability.
- Cost delta vs. original: low
- Anti-pattern check: pass if surfaces are clearly separated.

**Yes, and what if it could...** produce the first Activity suggestion without requiring the user to schedule it immediately?

- Serves: `jtbd-move-the-few-things-that-matter`
- Job elevation: The user sees how the Goal can become action, but capture/planning remains optional.
- New value: Clear next step without adding another required onboarding step.
- Cost delta vs. original: medium
- Anti-pattern check: pass; no forced commitment to dismiss.

## Frame Recommendation

Run the loop with an expanded frame: FTUX Goal+Arc onboarding.

The Activity-To-Identity work is the input model. The FTUX product model should be larger: one deterministic first-run flow that creates a concrete first Goal and an identity Arc together, then lands on the Arc detail screen.
