# Converge: object-creation-ux-system

## Chosen direction

Option B: Shared Grammar, Different Lenses.

## Why

FTUX needs more guidance because it creates first value and teaches the object model through experience. Direct Goal creation needs speed because the user is often already in motion. Direct Arc creation needs an identity-first path for users who arrive with a clear becoming direction, plus an activity-to-identity path for users who only know the concrete thing.

## Before / after

Today, each creation flow can ask different questions for historical reasons.

After this initiative, each flow uses the same creation grammar:

- Focus: what the user can name.
- Goal shape: what near-term progress should look like.
- Meaning: what matters most about it.
- Identity bridge: who this helps the user become.
- Resistance: where it gets hard.
- Support style: what help should feel like.

## Product rule

The questions do not have to be identical. The underlying grammar should be.

## Flow implications

FTUX:

- Full concrete-to-identity flow.
- Creates both first Goal and linked Arc.
- Lands on Arc detail.

Direct Goal creation:

- Minimal guided flow.
- Starts concrete and fast.
- Requires Arc attachment when the domain requires it.
- Asks a light meaning/identity question only when needed to create a better Goal or choose/create an Arc.

Direct Arc creation:

- Offers identity-first and activity-to-identity starts.
- Can create Arc only, or optionally seed a first Goal when concrete material is present.
- Should not force a Goal if the user is intentionally naming a long-term identity direction.

## Accepted trade-offs

- More documentation and tests are needed to prevent copy/config drift.
- Direct Goal creation may become slightly more guided than it is today.
- Direct Arc creation may need a clearer path choice instead of one hidden template.

## Rejected trade-offs

- Do not make all users complete the FTUX survey every time.
- Do not let direct Goal creation produce unanchored invalid Goals.
- Do not let direct Arc creation collapse into a generic goal-setting flow.

## Bet

We are betting that users will have more task success if Kwilt has one dependable creation grammar with shorter or longer exposure depending on context. If this proves too heavy, we should keep the grammar but reduce exposed questions in direct Goal creation first.

## Success signal

A user can create their first objects in FTUX, later create a Goal directly, and later create an Arc directly without feeling like Kwilt changed theories about what Goals and Arcs are.
