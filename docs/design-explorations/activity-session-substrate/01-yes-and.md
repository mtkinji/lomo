# Yes-And: Activity Session Substrate

Original idea: Activities are durable intentions; sessions are the time-bound attempts to live them out.

## Adjacencies

**Yes, and what if sessions became the bridge between Plan and Focus?**

- Serves: `jtbd-carry-intentions-into-action`
- Job elevation: The calendar block can carry the user to the exact Activity and Focus moment without starting Focus automatically.
- New value: Plan says when the attempt is intended; Focus records when the attempt actually began and ended.
- Cost delta vs. original: medium
- Anti-pattern check: pass, if Focus remains user-started and not schedule-triggered.

**Yes, and what if sessions gave Chapters better evidence without creating time tracking?**

- Serves: `jtbd-capture-and-find-meaning`
- Job elevation: Chapters can notice that the user returned to meaningful work several times without asking the user to log each attempt manually.
- New value: Reflection becomes more grounded in actual engagement patterns.
- Cost delta vs. original: medium to high
- Anti-pattern check: pass only if surfaced as humble observation, not totals or productivity scores.

**Yes, and what if the same substrate powered mobile and desktop continuity?**

- Serves: `jtbd-trust-this-app-with-my-life`
- Job elevation: A session started or planned in one surface can be understood by another without inventing surface-specific state.
- New value: Desktop can later resume "the current attempt" while mobile owns capture and calendar context.
- Cost delta vs. original: high
- Anti-pattern check: pass, if it does not become a command-center dashboard by default.

**Yes, and what if sessions clarified "planned, attempted, completed" without shaming drift?**

- Serves: `jtbd-move-the-few-things-that-matter`
- Job elevation: The system can distinguish a plan that moved, a plan that waited, and a plan that changed without treating waiting as failure.
- New value: More honest reflection and less brittle schedule adherence.
- Cost delta vs. original: medium
- Anti-pattern check: pass, if no missed-session alarms or red badges are introduced.

**Yes, and what if recurring Activities used sessions as occurrence evidence, not backlog copies?**

- Serves: `jtbd-trust-this-app-with-my-life`
- Job elevation: Recurring work can preserve real attempt history without spawning invisible piles of missed copies.
- New value: Aligns with the existing "one overdue occurrence" recurring posture.
- Cost delta vs. original: high; future-only
- Anti-pattern check: pass, if recurrence semantics remain separate and calm.

**Yes, and what if AI planning could propose sessions without owning commitment?**

- Serves: `jtbd-carry-intentions-into-action`
- Job elevation: AI could eventually suggest a few candidate attempts while the user stays in control of calendar writes.
- New value: Better planning assistance without background surprise edits.
- Cost delta vs. original: medium to high
- Anti-pattern check: pass only with preview/confirm and inspectable reasons.

**Yes, and what if session data became a cross-app primitive for the Kwilt suite?**

- Serves: `jtbd-trust-this-app-with-my-life`
- Job elevation: Budget, Desktop, and future Kwilt apps can record time-bound attempts without each inventing their own meaning model.
- New value: A coherent suite strategy: durable intention plus bounded engagement.
- Cost delta vs. original: high; strategy-only for now
- Anti-pattern check: pass, if the mobile app proves the primitive before suite generalization.

## Job Elevation

The original session idea solves scheduled to-do duplication. The elevated job is continuity across intention, attempt, engagement, and reflection. The value is not a visible Sessions feature; it is a quieter, more truthful application architecture.

## Frame Recommendation

**Run design-thinking-loop with the original frame** - "Activity Session Substrate" is the right strategic frame. It should produce a staged build plan: prove schedule sessions first, define the broader substrate carefully, and defer Focus/Chapter consumption until the semantics are stable.
