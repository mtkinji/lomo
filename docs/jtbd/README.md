# Jobs to be Done (JTBD)

This directory is one half of Kwilt's **demand-side source of truth**: a hierarchy of user jobs the app can satisfy, expressed in user voice, anchored to the identity-first philosophy in [docs/life-architecture-model.md](../life-architecture-model.md). The other half is [`docs/personas/`](../personas/), which names target audiences and representative people we are designing for.

## Why this exists

- **Forcing function for prioritization.** Every feature brief declares which audience/persona it is primarily for, the audience's `hero_jtbd:`, and which supporting or feature-level JTBDs it `serves:`. If a proposed feature can't cite a JTBD, it's either misframed or implies a missing JTBD — both useful signals.
- **Continuous pull on design decisions.** The user-level `design-thinking-loop` skill (at `~/.cursor/skills/design-thinking-loop/`) selects a target audience and named persona from `docs/personas/`, starts from that audience's hero JTBD, reads `docs/job-flows/` for current delivery, then reads this directory to assess which supporting or feature-level JTBDs are active for that person before solution generation. It discovers this taxonomy at the standard `docs/jtbd/_index.md` path.
- **Closed-loop validation.** The user-level `reflect-after-ship` skill updates each JTBD's `evidence`, `confidence`, and `realized_by` after a feature ships, so the taxonomy doesn't rot into write-once docs.

## How to read this directory

- Read audiences/personas and JTBDs as peer anchors: audiences name who we might serve, named personas make that audience concrete, and JTBDs name what becomes active for that person in a situation.
- Each audience has a hero JTBD in `docs/personas/`. The JTBD tree then descends into supporting and feature-level jobs that feature briefs can serve.
- Job flows in `docs/job-flows/` map JTBDs to product flow and delivery scores. They are product operating artifacts, not just design-thinking inputs.
- **Top-level JTBDs** live as files at the root (`see-who-im-becoming/index.md` or `<id>.md` if no children yet). These are the *personal-relevance bridge* jobs — why Kwilt is personally relevant to a user.
- **Sub-jobs** nest by directory: `<parent-id>/<child-id>.md`.
- **Special files**:
  - `_index.md` — flat machine-readable index of every JTBD id, regenerated by `npm run jtbd:lint`.
  - `_kwilt-context-primer.md` — condensed identity-first context the design-thinking-loop injects into the Diverge phase.
  - `README.md` (this file).

## How to author a JTBD

Every JTBD is a markdown file with YAML front-matter and a small body. Required shape:

```markdown
---
id: jtbd-<kebab-case-id>
title: "Help me <user-voice statement of what they want>"
parent: jtbd-<parent-id>  # null for top-level
level: top | mid | leaf
owner: andrew
last_reviewed: YYYY-MM-DD
confidence: hypothesis  # hypothesis | validated | retired
evidence: []            # free-form URLs / notes
realized_by: []         # feature brief ids that have shipped to serve this job
tags: []
---

## When this job is hot
[1-3 sentences of trigger context — what's the user doing or feeling when this job emerges]

## What "done" feels like for the user
[1-3 sentences of outcome in user voice — not feature output, lived experience]

## Sub-jobs
- jtbd-<sub-id>
- jtbd-<sub-id>
```

## Conventions

- **User voice, not feature voice.** "Help me see who I'm becoming," not "Display Arc progress."
- **Stable ids.** Once a JTBD is referenced by any feature brief or another JTBD, the id never changes. Retire by setting `confidence: retired`, don't delete or rename.
- **`confidence: hypothesis` is the default.** Bump to `validated` only after a shipped feature served it and we have evidence.
- **Don't put business goals here.** Acquisition, monetization, and growth metrics are not user JTBDs. They belong in the roadmap, not in this taxonomy.

## Cross-repo

JTBDs are canonical in the Kwilt mobile repo. The `kwilt-site` and `kwilt-desktop` repos symlink `docs/jtbd` → `../Kwilt/docs/jtbd`. feature briefs in any of the three repos reference the same ids.

## Linting

```bash
npm run jtbd:lint
```

Validates front-matter, parent references, and `serves:` references in feature briefs.
